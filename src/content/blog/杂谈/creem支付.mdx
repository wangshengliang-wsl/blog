最近在帮跨境团队折腾支付链路，本来想着换个服务商不就是签个合同、换个 SDK 的事儿嘛，结果踩完坑才发现清结算、合规、商户体验这些都得一起考虑。Creem 这个新玩家倒是给了我一个还不错的思路，趁着记忆还新鲜，把踩坑笔记整理出来，给同样在支付这摊浑水里扑腾的朋友参考参考。

## Creem 到底是个啥

简单说，Creem 把自己定位成「面向创作者和 SaaS 团队的全球收款堆栈」。听起来挺高大上，其实核心就是把合规账户、汇率、风控、代扣税这些破事打包在一起。跟传统 PSP 比起来，它更像是个"自带财务团队"的支付层，开箱就能对接 Stripe、Apple Pay、虚拟卡这些通道，还能自动生成税务报表。

说实话，一开始我也觉得这玩意儿是不是在画饼，用下来发现确实省了不少事。

## 用下来觉得不错的地方

**Onboarding 速度还行**：审核问的问题主要就是业务模式和跨境主体这些，准备好发票、股权架构图，基本一天就能搞定。比某些大厂动不动就拖你一周强多了。

**智能分账挺实用**：Dashboard 上配置规则，佣金、手续费、税费能自动拆分到不同虚拟子账户，不用再手工对账了。这个功能我们财务同事特别喜欢。

**FX 费率能锁价**：结算的时候可以锁价，不用再担心汇率突然跳水。虽然锁价窗口只有 15 分钟，但总比没有强。

**API 用起来顺手**：REST API + Webhook 的体感跟 Stripe 差不多，SDK 有 TypeScript、Python、Go 三套，文档写得也还算人话，不会像某些服务商那样写一堆废话。

## 怎么开始用

流程其实不复杂，但有几个地方容易卡住：

**第一步：提交材料**。营业执照、最终受益人信息、主要结算币种这些常规材料。官方建议把业务介绍做成 PPT 附件，审核的人看起来更直观。我们当时就随便写了个文档，结果被要求补充，所以建议一开始就做详细点。

**第二步：KYC/KYB**。通过 Notary 服务远程视频核身，法人得准备护照或驾照。视频通话大概 10 分钟，问的都是常规问题，不用太紧张。

**第三步：设置 Settlement Profile**。选主收款银行、币种、结算周期（T+1、T+3 可选），如果担心汇率波动可以启用自动对冲。我们选了 T+1，资金周转压力小一点。

**第四步：开沙箱**。Creem 会给一对 `client_id` / `client_secret`，还有测试卡和 webhook 密钥。沙箱环境跟生产基本一致，可以放心测试。

**第五步：联调过风控**。发 3~5 笔典型交易给审核团队，他们会验证你的风控策略。我们当时发了几个不同国家的订单，很快就过了。

## API 用起来什么样

调用方式跟 Stripe 差不多，直接看代码：

```bash
curl -X POST https://api.creem.io/payments \
  -H "Authorization: Bearer <API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{
        "amount": 12900,
        "currency": "USD",
        "capture": true,
        "description": "Pro 年度订阅",
        "customer": {
          "email": "customer@example.com",
          "country": "US"
        },
        "metadata": {
          "plan": "pro_yearly",
          "source": "web"
        }
      }'
```

响应里会返回 `status`, `net_amount`, `fx_rate` 这些字段，直接扔到财务系统就行。`net_amount` 是扣完手续费后的金额，不用自己再算一遍。

## 两种对接方式

### Hosted Checkout（适合快速上线）

如果你急着上线，用这个最快。Dashboard 上配置一下品牌色、Logo、必填字段，前端引入一段 JS 就能打开 Creem 托管的支付页。

好处是 PCI 责任基本都在 Creem 那边，你不用管。坏处是流程定制空间有限，想改个按钮样式都得看他们脸色。

### API + Webhook（适合深度定制）

我们选的是这个，前端表单和支付指令自己控制，Creem 只管网关和结算。

Webhook 事件有 `payment.succeeded`、`refund.processed`、`payout.created` 这些，记得用 `X-Creem-Signature` 校验一下来源，别被钓鱼了。

如果要做 Split Payment（分账），创建订单的时候得带上 `payout_rules`，不然默认全额进主账户。我们当时没注意这个，结果第一笔订单的钱全进主账户了，财务同事还纳闷怎么没分账。

```ts
// Next.js 处理 webhook 的示例
export async function POST(req: Request) {
  const payload = await req.text()
  verifySignature(payload, req.headers.get('x-creem-signature'))
  const event = JSON.parse(payload)

  if (event.type === 'payment.succeeded') {
    await markOrderPaid(event.data.referenceId)
  }

  return new Response('ok')
}
```

## 运营和风控的一些经验

**额度管理**：按国家/渠道设置单笔和每日上限，防止被那些刷卡团伙撞库。我们一开始没设上限，结果被刷了好几笔大额订单，虽然最后追回来了，但过程很折腾。

**风控白名单**：对忠实客户启用白名单，可以降低二次验证率，成功率会高不少。我们给 VIP 用户都加了白名单，支付成功率从 85% 提升到 92%。

**税务同步**：Creem 支持自动代扣 VAT/GST，但记得跟 ERP 对账，别重复申报了。我们财务同事一开始没注意，差点重复申报，还好及时发现。

**现金流预测**：结算周期可以配置成多档，结合 Dashboard 导出的 CSV 做滚动现金流模型，CFO 用起来还挺方便的。我们每周导一次数据，做下周的资金预测。

## 几个常见问题

**Q: 可以只用 FX 功能，不走它的支付通道吗？**

可以，通过 Virtual Account 方式接入就行，但还是得完成 KYB。我们一开始也这么想的，后来发现既然都要做 KYB，不如直接用它的支付通道，还能省点事。

**Q: 会不会和 Stripe 冲突？**

不冲突，Creem 还支持把 Stripe 当作子通道，用它的路由逻辑来做风控和对账。我们就是两个一起用的，Creem 负责结算，Stripe 负责部分地区的支付通道。

**Q: 结算周期能否缩短到 T+0？**

可以，但得单独申请，还要通过额外风控审核。通常只有那些留存资金比较多的 SaaS 才能申请下来。我们申请过，但被拒了，说资金量不够。

## 踩过的坑

**Webhook 重放问题**：第一次对接的时候没做重放保护，结果后台重复入账了。后来发现是 Creem 那边重试机制导致的，记得存 `event_id` 做幂等，别重复处理同一个事件。

**FX 锁价窗口太短**：默认只有 15 分钟，用户如果犹豫一下，价格就过期了。我们后来在订单页面加了实时刷新报价的功能，用户下单前会重新获取一次价格，避免金额对不上。

**跨币种退款有点坑**：跨币种退款的时候，会优先用原币种余额，不够了才从主账户换汇。这个逻辑财务同事一开始不知道，对账的时候发现金额对不上，折腾了半天才搞清楚。建议提前跟财务说清楚这个逻辑。

## 总结

如果你也在为多通道支付、跨国结算、税务申报这些破事头疼，Creem 确实值得试试。它不是万能的，但至少把那些繁琐的合规和对账细节做成了产品，能让你少折腾点，把时间花在业务上。

当然，每个业务场景不一样，我们的经验不一定适合所有人。如果你也在用 Creem，或者有其他支付服务商的踩坑经验，欢迎在评论区分享，大家一起避坑。
