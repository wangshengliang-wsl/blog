最近在寻找一个简单易用的部署平台，偶然发现了 Zeabur。用了一段时间后，发现它确实是个不错的 Vercel 替代品，特别是对国内开发者来说，访问速度和中文支持都更友好。趁着刚踩完坑，把 Zeabur 的核心概念和完整操作流程整理出来，给同样在找部署方案的朋友参考。

![Zeabur 平台界面](https://images.unsplash.com/photo-1558494949-ef010cbdcc31?auto=format&fit=crop&w=1200&q=80)

## Zeabur 是什么

Zeabur 是一个现代化的云部署平台，类似于 Vercel 和 Railway，但有一些自己的特色：

1. **一站式部署**：可以在一个项目中同时部署前端、后端、数据库、消息队列、对象存储等各种服务，不用分别部署到不同平台
2. **自动识别项目类型**：连接 GitHub 仓库后，Zeabur 会自动分析代码，识别框架类型（Next.js、React、Node.js、Python 等），并选择最适合的部署方式
3. **内置 CI/CD**：代码推送到 GitHub 后自动部署，无需手动配置 GitHub Actions
4. **按量计费**：采用按需付费模式，只为实际使用的资源付费，适合个人项目和小团队
5. **多环境隔离**：支持生产环境和预览环境，每个 Pull Request 都会自动创建预览部署
6. **快速域名**：每个服务都会自动生成一个 `*.zeabur.app` 的域名，也可以绑定自定义域名

整体来说，Zeabur 的目标是让部署变得像 `git push` 一样简单。

## Zeabur 核心概念

在开始使用之前，先理解几个核心概念：

### 项目（Project）

一个项目是容纳所有相关服务的容器。比如一个全栈应用，可以把前端、后端、数据库都放在同一个项目里。

### 服务（Service）

服务是项目中的单个应用或资源，比如：

- **Web 服务**：前端或后端应用（从 GitHub 部署）
- **数据库**：PostgreSQL、MySQL、MongoDB 等
- **存储**：Redis、对象存储等
- **其他**：消息队列、定时任务等

### 环境变量（Environment Variables）

每个服务都有自己的环境变量，可以在 Zeabur Dashboard 中配置。环境变量支持：

- 手动设置
- 从其他服务引用（比如数据库连接字符串）
- 加密存储

### 域名（Domain）

每个 Web 服务都会自动生成一个 `*.zeabur.app` 的域名，也可以绑定自定义域名。Zeabur 会自动处理 HTTPS 证书。

### 部署（Deployment）

每次代码推送或手动触发，都会创建一个新的部署。可以查看部署历史、回滚到之前的版本。

## 完整操作流程

### 第一步：注册和登录

1. 访问 [Zeabur 官网](https://zeabur.com)
2. 点击 **Sign Up**，使用邮箱注册
3. 收到验证邮件后，点击邮件中的链接完成验证
4. 登录后进入 Dashboard

### 第二步：创建项目

1. 在 Dashboard 点击 **Create Project**
2. 输入项目名称（比如 `my-app`）
3. 选择项目类型（个人或团队）
4. 点击 **Create** 完成创建

### 第三步：连接 GitHub

首次部署需要连接 GitHub：

1. 点击 **Deploy Service** 或 **Add Service**
2. 选择 **Deploy from GitHub**
3. 点击 **Connect GitHub**，授权 Zeabur 访问你的 GitHub 仓库
4. 选择要部署的仓库

### 第四步：部署服务

连接 GitHub 后，选择要部署的仓库：

1. **自动识别**：Zeabur 会自动分析代码，识别项目类型
   - Next.js → 自动配置构建命令和输出目录
   - React/Vite → 自动配置静态文件部署
   - Node.js → 自动识别入口文件
   - Python → 自动识别框架（Django、Flask 等）

2. **配置部署**：
   - **Root Directory**：如果项目在子目录，可以指定根目录
   - **Build Command**：构建命令（通常自动识别）
   - **Start Command**：启动命令（通常自动识别）
   - **Output Directory**：输出目录（静态文件部署）

3. **环境变量**：配置所需的环境变量

   ```bash
   # 示例：Next.js 项目
   NODE_ENV=production
   NEXT_PUBLIC_API_URL=https://api.example.com
   DATABASE_URL=postgresql://...
   ```

4. **点击 Deploy**：开始部署

部署过程中可以看到实时日志，通常几分钟就能完成。

### 第五步：配置数据库

如果需要数据库，可以在同一个项目中添加：

1. 在项目页面点击 **Add Service**
2. 选择 **Database** → 选择数据库类型（PostgreSQL、MySQL、MongoDB 等）
3. 配置数据库名称和版本
4. 点击 **Deploy**

数据库部署后，会自动生成连接字符串，可以在环境变量中引用：

```bash
# 在 Web 服务的环境变量中引用数据库
DATABASE_URL={{services.database.env.DATABASE_URL}}
```

### 第六步：配置自定义域名

1. 在服务页面点击 **Domains**
2. 点击 **Add Domain**
3. 输入你的域名（比如 `app.example.com`）
4. 按照提示配置 DNS：
   - 添加 CNAME 记录：`app` → `your-service.zeabur.app`
   - 或者添加 A 记录（如果支持）
5. 等待 DNS 生效（通常几分钟到几小时）
6. Zeabur 会自动申请和配置 SSL 证书

### 第七步：配置环境变量

环境变量可以在多个地方配置：

#### 方法一：在 Dashboard 中配置

1. 进入服务页面
2. 点击 **Environment Variables**
3. 点击 **Add Variable**
4. 输入变量名和值
5. 点击 **Save**

#### 方法二：引用其他服务的环境变量

如果数据库和 Web 服务在同一个项目，可以直接引用：

```bash
# 引用数据库的连接字符串
DATABASE_URL={{services.database.env.DATABASE_URL}}

# 引用其他服务的 URL
API_URL={{services.backend.env.ZEABUR_URL}}
```

#### 方法三：使用 .env 文件（本地开发）

Zeabur 不支持直接上传 `.env` 文件，但可以在本地开发时使用，部署时在 Dashboard 配置。

### 第八步：配置自动部署

默认情况下，推送到主分支会自动部署到生产环境。可以配置：

1. 进入服务页面
2. 点击 **Settings** → **Deployments**
3. 配置部署规则：
   - **Auto Deploy**：主分支自动部署
   - **Preview Deployments**：Pull Request 自动创建预览环境
   - **Branch Deployments**：指定分支自动部署

## 实际部署示例

### 示例 1：部署 Next.js 应用

假设有一个 Next.js 项目，结构如下：

```
my-nextjs-app/
├── package.json
├── next.config.js
└── src/
    └── app/
```

部署步骤：

1. **连接 GitHub**：选择仓库 `my-nextjs-app`
2. **自动识别**：Zeabur 识别为 Next.js 项目
3. **配置环境变量**：
   ```bash
   NODE_ENV=production
   NEXT_PUBLIC_API_URL=https://api.example.com
   ```
4. **部署**：点击 Deploy，等待完成

Zeabur 会自动：

- 运行 `npm install` 或 `pnpm install`
- 运行 `npm run build`
- 启动 `npm start` 或 `node server.js`

### 示例 2：部署全栈应用（前端 + 后端 + 数据库）

假设有一个全栈项目：

```
my-fullstack-app/
├── frontend/        # Next.js 前端
├── backend/         # Node.js 后端
└── package.json
```

部署步骤：

1. **部署数据库**：
   - 添加 PostgreSQL 服务
   - 记录连接字符串

2. **部署后端**：
   - 选择仓库，Root Directory 设置为 `backend`
   - 配置环境变量：
     ```bash
     DATABASE_URL={{services.database.env.DATABASE_URL}}
     PORT=3001
     ```
   - 部署

3. **部署前端**：
   - 选择仓库，Root Directory 设置为 `frontend`
   - 配置环境变量：
     ```bash
     NEXT_PUBLIC_API_URL={{services.backend.env.ZEABUR_URL}}
     ```
   - 部署

4. **配置域名**：
   - 前端绑定 `app.example.com`
   - 后端可以绑定 `api.example.com`（可选）

### 示例 3：使用 Dockerfile 部署

如果项目有 `Dockerfile`，Zeabur 会自动使用 Docker 部署：

```dockerfile
# Dockerfile 示例
FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .
RUN npm run build

EXPOSE 3000

CMD ["npm", "start"]
```

Zeabur 会自动：

1. 构建 Docker 镜像
2. 运行容器
3. 暴露端口

## 环境变量最佳实践

### 1. 敏感信息加密

Zeabur 会自动加密存储环境变量，但建议：

- 不要在代码中硬编码敏感信息
- 使用环境变量引用其他服务
- 定期轮换密钥和 token

### 2. 环境隔离

不同环境使用不同的环境变量：

- **Production**：生产环境变量
- **Preview**：预览环境变量（PR 部署）

### 3. 引用其他服务

使用 `{{services.service-name.env.VARIABLE_NAME}}` 引用其他服务的环境变量：

```bash
# 引用数据库
DATABASE_URL={{services.postgres.env.DATABASE_URL}}

# 引用 Redis
REDIS_URL={{services.redis.env.REDIS_URL}}

# 引用其他 Web 服务
API_URL={{services.backend.env.ZEABUR_URL}}
```

## 监控和日志

### 查看日志

1. 进入服务页面
2. 点击 **Logs** 标签
3. 可以看到实时日志和历史日志
4. 支持搜索和过滤

### 查看指标

1. 进入服务页面
2. 点击 **Metrics** 标签
3. 可以看到：
   - CPU 使用率
   - 内存使用率
   - 网络流量
   - 请求数

### 查看部署历史

1. 进入服务页面
2. 点击 **Deployments** 标签
3. 可以看到所有部署记录
4. 可以回滚到之前的版本

## 常见问题和踩坑

### 问题 1：部署失败，提示构建错误

**原因**：构建命令或依赖安装失败

**解决方案**：

- 检查构建日志，找到具体错误
- 确认 `package.json` 中的脚本正确
- 检查 Node.js 版本是否匹配（可以在 `package.json` 中指定 `engines.node`）
- 确认所有依赖都已正确安装

### 问题 2：环境变量未生效

**原因**：环境变量配置错误或未重启服务

**解决方案**：

- 检查环境变量名称是否正确（大小写敏感）
- 确认环境变量已保存
- 重启服务（重新部署）
- 检查是否在正确的环境（Production/Preview）中配置

### 问题 3：数据库连接失败

**原因**：数据库 URL 配置错误或网络问题

**解决方案**：

- 确认数据库服务已启动
- 检查数据库连接字符串格式
- 确认使用了正确的环境变量引用语法：`{{services.database.env.DATABASE_URL}}`
- 检查数据库是否允许外部连接（Zeabur 会自动配置）

### 问题 4：自定义域名无法访问

**原因**：DNS 配置错误或未生效

**解决方案**：

- 检查 DNS 记录是否正确配置
- 确认 CNAME 记录指向正确的 Zeabur 域名
- 等待 DNS 生效（可能需要几小时）
- 检查域名是否已通过验证

### 问题 5：预览环境无法访问

**原因**：预览部署配置错误

**解决方案**：

- 确认已启用 Preview Deployments
- 检查 Pull Request 是否已创建
- 确认预览环境的域名可以访问
- 检查预览环境的环境变量是否正确

### 问题 6：服务启动失败

**原因**：启动命令错误或端口配置错误

**解决方案**：

- 检查 Start Command 是否正确
- 确认服务监听的是 `0.0.0.0` 而不是 `localhost`
- 确认端口是否正确（Zeabur 会自动设置 `PORT` 环境变量）
- 检查日志中的错误信息

## 最佳实践

### 1. 项目结构

建议将相关服务放在同一个项目中：

- 前端 + 后端 + 数据库 → 一个项目
- 微服务架构 → 每个服务一个项目，或按业务分组

### 2. 环境变量管理

- 使用环境变量引用，避免硬编码
- 敏感信息使用加密存储
- 不同环境使用不同的变量值

### 3. 部署策略

- 主分支自动部署到生产环境
- Pull Request 自动创建预览环境
- 重要更新前先在预览环境测试

### 4. 监控和告警

- 定期查看日志和指标
- 设置告警规则（如果支持）
- 监控服务健康状态

### 5. 成本控制

- 定期检查资源使用情况
- 停止不需要的服务
- 使用预览环境测试，避免频繁部署生产环境

## 总结

Zeabur 是一个非常适合个人开发者和小团队的部署平台，特别是对国内开发者来说，中文支持和访问速度都是优势。核心优势在于：

1. **简单易用**：连接 GitHub 后自动识别和部署，几乎零配置
2. **一站式服务**：前端、后端、数据库都可以在同一个平台管理
3. **按量计费**：只为实际使用的资源付费，成本可控
4. **自动 CI/CD**：代码推送自动部署，无需手动配置

如果你也在寻找一个简单易用的部署平台，特别是需要部署全栈应用或需要数据库的场景，Zeabur 是个不错的选择。相比 Vercel，它在数据库和中文支持方面更有优势；相比 Railway，它的界面和文档对中文用户更友好。

如果你也在用 Zeabur，或者有其他部署平台的踩坑经验，欢迎在评论区分享，大家一起避坑。
